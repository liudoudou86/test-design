# 测试用例生成指南

本文档提供生成测试用例时穷举所有测试点的详细方法，确保测试用例覆盖完整。

## 一、测试用例生成方法总览

| 方法               | 适用场景                             | 优点                           | 缺点                     |
| ------------------ | ------------------------------------ | ------------------------------  | -------------------------- |
| 等价类划分法       | 输入型功能，如表单、API参数           | 减少用例数量，提高效率         | 需要准确划分等价类         |
| 边界值分析法       | 输入有明确范围限制                   | 发现边界附近的缺陷             | 仅适用于有边界的情况       |
| 场景法             | 流程型功能，如购物流程、审批流程     | 覆盖业务流程，贴近实际使用     | 用例数量可能较多           |
| 因果图/决策表法   | 复杂逻辑判断，如折扣规则、权限控制   | 系统覆盖所有组合               | 构建复杂，需要专业知识     |
| 错误猜测法         | 补充异常场景                         | 基于经验，发现隐藏缺陷         | 依赖测试人员经验           |

## 二、等价类划分法（#equivalence）

### 2.1 方法原理

等价类划分法是将输入数据按照"等价"原则划分为若干个子集（等价类），假设同一等价类中的数据对发现缺陷的效果相同，只需从每个等价类中选取少量数据作为测试数据。

### 2.2 等价类类型

| 类型           | 说明                               | 示例                                       |
| -------------- | ---------------------------------- | ------------------------------------------ |
| 有效等价类     | 符合需求规格说明的输入             | 用户名长度6-20字符，"testuser"是有效等价类 |
| 无效等价类     | 不符合需求规格说明的输入           | 用户名长度<6字符，"test"是无效等价类       |

### 2.3 划分原则

1. **按输入条件划分**：如果输入条件规定了输入值的范围（如1-100），则划分有效等价类（1-100）和两个无效等价类（<1和>100）
2. **按输入集合划分**：如果输入条件规定了输入值的集合（如性别为男/女），则划分两个有效等价类（男、女）和一个无效等价类（其他）
3. **按布尔值划分**：如果输入条件是布尔值（如是/否），则划分两个有效等价类（是、否）
4. **按规则划分**：如果输入条件规定了规则（如邮箱格式），则划分有效等价类（符合规则）和无效等价类（不符合规则）

### 2.4 应用步骤

1. **分析需求**：理解输入条件，明确有效和无效的边界
2. **划分等价类**：按照划分原则，列出所有有效和无效等价类
3. **设计用例**：从每个有效等价类中选取一个或多个代表值，为每个无效等价类设计一个用例

## 三、边界值分析法（#boundary）

### 3.1 方法原理

边界值分析法是对等价类划分法的补充，针对输入或输出范围的边界条件进行测试。经验表明，缺陷更容易发生在边界上，因此重点测试边界值。

### 3.2 边界值类型

| 类型                 | 说明                   | 示例                               |
| -------------------- | ---------------------- | ---------------------------------- |
| 上边界               | 范围的最大值           | 如1-100，上边界为100               |
| 下边界               | 范围的最小值           | 如1-100，下边界为1                 |
| 略大于上边界         | 上边界+最小增量         | 如1-100，略大于上边界为101         |
| 略小于下边界         | 下边界-最小增量         | 如1-100，略小于下边界为0           |
| 刚好在上边界之内     | 上边界-最小增量         | 如1-100，刚好在上边界之内为99       |
| 刚好在下边界之内     | 下边界+最小增量         | 如1-100，刚好在下边界之内为2         |

### 3.3 选取原则

对于每个输入或输出范围，选取以下边界值：
- 刚好等于下边界
- 刚好大于下边界
- 刚好小于下边界
- 刚好等于上边界
- 刚好小于上边界
- 刚好大于上边界

### 3.4 应用步骤

1. **确定边界**：分析需求，确定输入或输出范围的上下边界
2. **选取边界值**：按照选取原则，列出所有需要测试的边界值
3. **设计用例**：为每个边界值设计测试用例

## 四、场景法（#scenario）

### 4.1 方法原理

场景法是通过模拟用户的实际使用场景，设计测试用例。场景法关注业务流程，覆盖正常流程和异常流程。

### 4.2 场景类型

| 类型       | 说明                                 | 示例                                               |
| ---------- | ------------------------------------ | -------------------------------------------------- |
| 正常场景   | 按照预期流程执行，每一步都成功       | 浏览商品→加入购物车→提交订单→支付成功             |
| 异常场景   | 流程中某一步失败，验证系统处理能力   | 浏览商品→加入购物车→提交订单→支付失败             |
| 边界场景   | 流程中某个环节处于边界条件           | 购买数量为库存上限、支付金额为最大限额           |
| 并发场景   | 多个用户同时操作                     | 多个用户同时购买同一商品                           |
| 性能场景   | 流程在特定性能条件下执行             | 弱网环境下提交订单                                 |

### 4.3 场景识别方法

1. **流程图分析**：根据需求绘制业务流程图，识别所有可能的路径
2. **用户故事分析**：根据用户故事，模拟用户操作路径
3. **状态转换分析**：分析系统状态转换，识别不同的场景
4. **前置后置条件分析**：分析每个步骤的前置条件和后置条件

### 4.4 应用步骤

1. **梳理流程**：分析需求，梳理完整的业务流程
2. **识别场景**：识别正常场景、异常场景、边界场景等
3. **设计用例**：为每个场景设计测试用例，明确步骤序列和预期结果

## 五、因果图/决策表法（#decision）

### 5.1 方法原理

因果图/决策表法用于处理输入条件的组合，特别是当多个输入条件之间存在约束关系时。通过分析输入条件之间的因果关系和约束关系，设计测试用例。

### 5.2 因果图基本符号

| 符号   | 说明                               |
| ------ | ---------------------------------- |
| 恒等   | 若原因A成立，则结果B成立           |
| 非     | 若原因A成立，则结果B不成立         |
| 或     | 若原因A或B任一成立，则结果C成立   |
| 与     | 若原因A和B都成立，则结果C成立     |

### 5.3 约束关系

| 约束类型 | 说明                               |
| -------- | ---------------------------------- |
| 互斥     | 多个原因不能同时为真               |
| 包含     | 多个原因至少有一个为真             |
| 唯一     | 多个原因有且只有一个为真           |
| 要求     | 若原因A为真，则原因B必须为真       |
| 掩盖     | 若原因A为真，则结果B必须为真       |

### 5.4 决策表结构

| 条件桩 | 条件项1 | 条件项2 | ... |
|--------|----------|----------|-----|
| 动作桩 | 动作项1 | 动作项2 | ... |
|        | 结果1   | 结果2   | ... |

### 5.5 应用步骤

1. **分析条件**：列出所有输入条件和可能的取值
2. **分析动作**：列出所有可能的输出或系统动作
3. **建立决策表**：根据条件和动作建立决策表
4. **简化决策表**：合并重复或无关的规则
5. **设计用例**：为每个规则设计测试用例

## 六、错误猜测法

### 6.1 方法原理

错误猜测法基于测试人员的经验和直觉，推测系统中可能存在的错误，针对性地设计测试用例。错误猜测法是一种补充方法，通常与其他方法结合使用。

### 6.2 常见错误场景

| 错误类型     | 典型场景                                                   |
| ------------ | ---------------------------------------------------------- |
| 输入错误     | 输入特殊字符、SQL注入、XSS攻击                             |
| 边界错误     | 边界值附近的异常、整数溢出                                 |
| 逻辑错误     | 条件判断错误、循环错误、状态转换错误                       |
| 性能错误     | 大数据量处理、高并发、弱网环境                             |
| 安全错误     | 权限越权、数据泄露、会话劫持                               |
| 兼容性错误   | 不同浏览器/设备表现不一致                                 |
| 数据错误     | 空值、重复数据、非法数据                                   |

### 6.3 应用建议

- 基于历史缺陷分析，总结常见错误类型

## 七、测试用例详细程度标准

根据需求，测试用例采用中等详细程度撰写。

### 7.1 中等详细程度的定义

- **关键步骤详细**：涉及核心业务逻辑、数据验证、状态转换的步骤需要详细描述
- **常规步骤简化**：常规操作（如点击按钮、输入数据）可简化描述
- **重点突出**：重点描述测试目的、关键数据和预期结果

### 7.2 详细程度对比

| 详细程度 | 步骤描述                                                         | 适用场景                         |
| -------- | ---------------------------------------------------------------- | -------------------------------- |
| 详细     | 每个操作都详细描述，包含具体数据、位置、操作方式                 | 复杂功能、新功能、高风险场景     |
| 中等     | 关键步骤详细，常规步骤简化                                       | 大部分测试用例                   |
| 简洁     | 仅描述核心步骤，适合快速执行                                     | 冒烟测试、回归测试               |

### 7.3 中等详细程度示例

**详细程度示例：登录功能**

| 详细程度 | 测试步骤                                                                                           |
| -------- | -------------------------------------------------------------------------------------------------- |

| 中等 | 1.打开登录页面<br>2.输入用户名：`testuser@example.com`<br>3.输入密码：`Password123`<br>4.点击登录按钮 |
| 简洁     | 输入正确的用户名和密码，点击登录                                                                   |

**推荐**：使用中等详细程度。

## 十一、AI功能测试用例生成特殊考虑

对于AI相关功能（如推荐、风控、翻译等），除了上述通用方法外，还需考虑以下特殊测试方法（可引用`references/test_strategy.md`）：

### 11.1 数据分布等价类划分

将输入数据按特征分布划分等价类，测试不同数据分布下AI输出的一致性。

### 11.2 鲁棒性与对抗性测试

- **鲁棒性测试**：输入微小扰动数据，测试AI输出稳定性
- **对抗性测试**：生成对抗样本，测试AI抗攻击能力

### 11.3 可解释性测试

验证AI决策的可解释性，确保决策依据可追溯。

### 11.4 数据漂移测试

模拟数据漂移场景，测试AI模型的性能衰减情况。

### 11.5 变质测试（Metamorphic Testing）

针对AI非确定性输出的特性，设计输入变换规则，验证输出是否满足预设关系。

### 11.6 偏见与公平性测试

验证AI决策是否存在性别、年龄、地域等偏见。

## 十二、测试用例生成最佳实践

1. **结合多种方法**：不要局限于单一方法，根据需求特点组合使用多种测试方法
2. **优先级分层**：根据业务重要性和风险等级，合理分配测试用例的优先级
3. **持续优化**：根据测试结果和缺陷反馈，持续优化测试用例
4. **可维护性**：测试用例应易于理解和维护，避免过度复杂的步骤
5. **可追溯性**：测试用例应与需求、用户故事建立可追溯关系
6. **评审机制**：定期评审测试用例，确保覆盖度和质量
7. **工具辅助**：利用测试管理工具提高测试用例的管理和执行效率
