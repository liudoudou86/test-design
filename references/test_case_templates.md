# 测试用例模板

本文档定义了生成测试用例时使用的标准格式、不同测试类型的模板和Excel/CSV输出规范。

## 一、测试用例标准格式

### 1.1 字段定义

| 字段名   | 说明                         | 必填 | 示例                                                           |
| -------- | ---------------------------- | ---- | -------------------------------------------------------------- |
| ID       | 唯一标识符，命名规范见下文   | 是   | TC-FUNC-LOGIN-001                                              |
| 优先级   | 测试用例的执行优先级         | 是   | P0/P1/P2/P3                                                    |
| 测试类型 | 测试用例所属类型             | 是   | 功能测试                                                       |
| 用例描述 | 简明扼要地描述测试目的       | 是   | 验证使用正确的用户名和密码可以成功登录                         |
| 前置条件 | 执行测试用例前需要满足的条件 | 是   | 用户已注册并激活账号                                           |
| 测试步骤 | 执行测试的具体步骤           | 是   | 1.打开登录页面<br>2.输入用户名<br>3.输入密码<br>4.点击登录按钮 |
| 预期结果 | 预期的输出或系统行为         | 是   | 跳转到首页，显示用户信息                                       |
| 实际结果 | 实际执行后的结果             | 否   | 执行后填写                                                     |
| 备注     | 补充说明或注意事项           | 否   | 特殊数据说明                                                   |

### 1.2 优先级定义

| 优先级 | 说明                     | 使用场景                           |
| ------ | ------------------------ | ---------------------------------- |
| P0     | 最高优先级，必须测试     | 核心业务流程、关键功能、高风险场景 |
| P1     | 高优先级，必须测试       | 重要功能、常用场景、边界条件       |
| P2     | 中等优先级，建议测试     | 辅助功能、异常场景、兼容性         |
| P3     | 低优先级，时间允许时测试 | 边缘场景、UI细节、体验优化         |

## 二、测试用例ID命名规范

### 2.1 命名格式

```
TC-{类型缩写}-{模块缩写}-{序号}
```

### 2.2 类型缩写

| 测试类型   | 缩写 |
| ---------- | ---- |
| 功能测试   | FUNC |
| UI/UX测试  | UIUX |
| 兼容性测试 | COMP |
| 性能测试   | PERF |
| 安全测试   | SECU |
| 无障碍测试 | ACCE |

### 2.3 模块缩写

根据具体项目自定义，建议使用拼音首字母或英文缩写，2-4个字符。例如：
- 登录模块：LOGIN
- 用户管理：USER
- 订单管理：ORDER
- 支付功能：PAY
- 商品管理：PROD

### 2.4 示例

- `TC-FUNC-LOGIN-001`：登录模块功能测试用例001
- `TC-UIUX-PROD-015`：商品模块UI/UX测试用例015
- `TC-COMP-PAY-003`：支付模块兼容性测试用例003

## 三、不同测试类型的用例模板

### 3.1 功能测试用例模板（#functional）

**适用场景**：验证系统功能是否按需求规格说明正确工作。

**用例特点**：
- 关注功能正确性、数据准确性、错误处理
- 关键步骤详细描述，常规步骤可简化
- 预期结果需明确具体的系统行为或输出

**示例**：

| ID                | 优先级 | 测试类型 | 用例描述                               | 前置条件             | 测试步骤                                                                   | 预期结果                                           | 实际结果 | 备注 |
| ----------------- | ------ | -------- | -------------------------------------- | -------------------- | -------------------------------------------------------------------------- | -------------------------------------------------- | -------- | ---- |
| TC-FUNC-LOGIN-001 | P0     | 功能测试 | 验证使用正确的用户名和密码可以成功登录 | 用户已注册并激活账号 | 1.打开登录页面<br>2.输入正确的用户名<br>3.输入正确的密码<br>4.点击登录按钮 | 跳转到首页，页面顶部显示用户昵称，登录状态为已登录 |          |      |

### 3.2 UI/UX测试用例模板（#ui-ux）

**适用场景**：验证界面设计符合规范，用户交互符合直觉。

**用例特点**：
- 关注界面一致性、易用性、交互反馈
- 步骤相对简洁，重点描述界面元素和行为
- 预期结果侧重视觉和交互体验

**示例**：

| ID                | 优先级 | 测试类型  | 用例描述               | 前置条件     | 测试步骤                   | 预期结果                             | 实际结果 | 备注 |
| ----------------- | ------ | --------- | ---------------------- | ------------ | -------------------------- | ------------------------------------ | -------- | ---- |

### 3.3 兼容性测试用例模板（#compatibility）

**适用场景**：验证软件在不同环境下都能正常工作。

**用例特点**：
- 关注跨浏览器、跨平台、跨设备、响应式设计
- 步骤简洁，重点描述环境差异对系统的影响
- 预期结果需明确在不同环境下的一致性表现

**示例**：

| ID                | 优先级 | 测试类型   | 用例描述                             | 前置条件           | 测试步骤                                                                     | 预期结果                             | 实际结果 | 备注 |
| ----------------- | ------ | ---------- | ------------------------------------ | ------------------ | ---------------------------------------------------------------------------- | ------------------------------------ | -------- | ---- |
| TC-COMP-LOGIN-010 | P1     | 兼容性测试 | 验证登录功能在Chrome浏览器上的兼容性 | 已安装Chrome浏览器 | 1.使用Chrome浏览器访问登录页面<br>2.输入正确的用户名和密码<br>3.点击登录按钮 | 登录成功，页面布局正常，无错位或错乱 |          |      |

**环境组合参考**：

| 测试维度 | 覆盖范围                                           |
| -------- | -------------------------------------------------- |
| 浏览器   | Chrome、Firefox、Safari、Edge（最新及前1-2个版本） |
| 操作系统 | Windows 10/11、macOS、iOS、Android                 |
| 设备     | PC、平板、手机（不同屏幕尺寸和分辨率）             |
| 网络环境 | 4G、5G、WiFi、弱网环境                             |

**用例特点**：
- 关注响应时间、吞吐量、错误率、资源使用率
- 需明确测试指标和性能基准
- 预期结果需量化性能指标

**示例**：

| ID                | 优先级 | 测试类型 | 用例描述                              | 前置条件       | 测试步骤                                                                    | 预期结果                                         | 实际结果 | 备注 |

## 四、中文测试用例撰写规范

### 4.1 用例标题撰写规范

- **简洁明确**：标题应简明扼要地描述测试目的，通常不超过30字
- **使用动宾结构**：如"验证"、"测试"、"检查"等动词开头
- **避免模糊表述**：避免使用"可能"、"大概"等模糊词汇
- **包含关键信息**：明确测试对象、测试条件、测试结果

**正确示例**：
- 验证使用正确的用户名和密码可以成功登录
- 验证用户名为空时登录按钮不可点击
- 检查登录失败时的错误提示信息

**错误示例**：
- 测试登录功能（过于模糊）
- 登录可能成功（使用了模糊词汇）
- 验证用户名和密码（缺少测试条件）

### 4.2 测试步骤撰写规范

- **步骤编号**：使用阿拉伯数字编号，每行一个步骤
- **动词开头**：每个步骤以动作动词开头，如"输入"、"点击"、"选择"等
- **明确具体**：操作步骤应具体明确，避免歧义
- **关键步骤详细，常规步骤简化**：符合中等详细程度的要求
- **包含关键数据**：如果测试依赖特定数据，需在步骤中说明

**正确示例**：
1. 打开登录页面
2. 在用户名输入框输入：`testuser@example.com`
3. 在密码输入框输入：`Password123`
4. 点击"登录"按钮

**错误示例**：
1. 输入用户名和密码（两个操作合并，不够具体）
2. 输入密码（缺少关键数据，应说明输入什么密码）
3. 填写表单（过于笼统）

### 4.3 预期结果撰写规范

- **具体明确**：预期结果应具体明确，可验证、可观察
- **量化指标**：如适用，使用量化指标（如响应时间<2秒）
- **避免主观判断**：避免使用"正常"、"正确"等主观词汇
- **包含系统行为**：描述系统应该表现出的具体行为
- **包含界面提示**：如有界面提示，需描述提示内容和位置

**正确示例**：
- 跳转到首页，页面顶部显示用户昵称"张三"
- 登录按钮显示为灰色，不可点击
- 弹出错误提示："用户名不能为空"，提示框位于页面顶部，3秒后自动消失
- 平均响应时间<2秒，95%请求响应时间<3秒

**错误示例**：
- 登录成功（主观判断，不够具体）
- 系统正常响应（"正常"过于主观）
- 有错误提示（缺少提示内容和位置）

## 五、Excel/CSV输出格式规范

### 5.1 Excel输出格式

**文件命名规范**：
```
{项目名称}_{模块名称}_测试用例_{版本}_{日期}.xlsx
```
例如：`电商系统_登录模块_测试用例_v1.0_20240101.xlsx`

**Sheet结构**：
- Sheet1：测试用例总览（包含所有类型的测试用例）
- Sheet2（可选）：功能测试用例
- Sheet3（可选）：性能测试用例
- 其他Sheet按测试类型分类

**表头格式**：
第一行为表头，使用加粗字体，背景色为浅灰色。

**单元格格式**：
- 用例ID：文本格式，左对齐
- 用例标题：文本格式，自动换行
- 优先级：居中对齐
- 测试步骤：自动换行，行高自动调整
- 预期结果：自动换行，行高自动调整

### 5.2 CSV输出格式

**文件命名规范**：
```
{项目名称}_{模块名称}_测试用例_{版本}_{日期}.csv
```
例如：`电商系统_登录模块_测试用例_v1.0_20240101.csv`

**编码格式**：UTF-8 with BOM（确保Excel能正确显示中文）

**分隔符**：半角逗号（`,`）

**引用符**：双引号（`"`），用于包含逗号或换行符的字段

**表头行**：第一行为表头字段名

**示例**：
```csv
ID,优先级,测试类型,用例描述,前置条件,测试步骤,预期结果,实际结果,备注
TC-FUNC-LOGIN-001,P0,功能测试,验证使用正确的用户名和密码可以成功登录,用户已注册并激活账号,"1.打开登录页面
2.输入正确的用户名
3.输入正确的密码
4.点击登录按钮",跳转到首页，页面顶部显示用户昵称，登录状态为已登录,,
```

## 六、Test Case Hierarchy Design（测试用例分层设计）

### 6.1 设计理念

来源于 deanpeters/Product-Manager-Skills（1.6k Star），将测试用例按重要性和覆盖范围分为三层：

| 层级 | 定义 | 特点 | 示例（登录功能） |
|------|------|------|------------------|
| **核心用例** | 主流程可用，验证核心功能正常 | P0优先级，覆盖率最高 | 账号密码正确登录成功 |
| **边界用例** | 参数/场景边界条件 | P1优先级，发现边界缺陷 | 密码长度正好8位、11位手机号 |
| **异常用例** | 异常输入、操作、环境 | P1/P2优先级，验证容错能力 | 账号为空、密码错误、验证码过期、多设备同时登录 |

### 6.2 实操方法

1. **选一个功能**（如登录）
2. **列出核心路径**：正常登录成功流程 → 核心用例
3. **找边界**：输入长度边界、状态边界 → 边界用例
4. **找异常**：空值、错误值、冲突场景 → 异常用例
5. **标优先级**：核心用例P0，边界用例P1，异常用例P1/P2

### 6.3 分层输出模板

```
用例分层清单 - [功能名称]
├─ 核心用例（P0）
│   ├─ TC-XXX-001: 验证正常流程
│   └─ TC-XXX-002: 验证核心分支
├─ 边界用例（P1）
│   ├─ TC-XXX-010: 验证输入下边界
│   ├─ TC-XXX-011: 验证输入上边界
│   └─ TC-XXX-012: 验证状态边界
└─ 异常用例（P1/P2）
    ├─ TC-XXX-020: 验证空输入
    ├─ TC-XXX-021: 验证错误输入
    └─ TC-XXX-022: 验证环境异常
```

## 七、测试场景模拟

### 7.1 设计理念

来源于 tester-skills/Test-Scenario-Generator（测试专属），解决面试或设计时场景零散、无框架的问题。

### 7.2 输入与输出

**输入**：待测功能（如「购物车结算」）

**输出**：完整的场景清单，包含：
- **正常场景**：完整用户路径（选品→结算→支付→确认）
- **异常场景**：每步骤的失败情况（库存不足、支付失败、地址无效）
- **边界场景**：数量边界、金额边界（最大商品数、最小支付金额）

### 7.3 场景拆解框架

| 维度 | 内容 | 示例（购物车结算） |
|------|------|-------------------|
| **核心逻辑梳理** | 功能的主要业务流程 | 选择商品→加入购物车→调整数量→提交订单→支付 |
| **用户路径拆解** | 完整操作步骤 | 浏览商品页→加入购物车→查看购物车→去结算→选择地址→选择支付方式→确认支付 |
| **隐性场景排查** | 不易察觉但重要的场景 | 购物车商品失效、优惠券叠加规则、库存实时变化 |
| **异常模板补充** | 标准化异常情况 | 商品售罄、库存不足、支付超时、地址不可用 |

### 7.4 场景清单输出模板

```
测试场景清单 - [功能名称]

【正常场景】
1. 场景1：完整购物流程
   路径：选品→加入购物车→结算→支付→确认
   预期：订单创建成功，支付成功
```
